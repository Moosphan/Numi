# Numi iOS SwiftUI Backlog

版本：0.1  
日期：2026-06-18  
范围：纯 iOS 原生 SwiftUI 首版 MVP  
依据：PRD、`style.md`、`docs/design`、`docs/tech/ios-swiftui-technical-solution.md`

## 1. 优先级定义

- P0：MVP 必须完成，不完成无法形成可用记账闭环。
- P1：首版强烈建议完成，影响体验完整度但不阻塞核心记账。
- P2：后续版本扩展。

### 1.1 估算规则

估算使用轻量 Story Point：

- 1：半天内可完成，主要是配置、文案、简单组件或小测试。
- 2：1 天左右，有明确输入输出。
- 3：1-2 天，需要 ViewModel、状态或少量持久化。
- 5：2-4 天，涉及完整页面、Repository 或复杂组件。
- 8：4-6 天，涉及跨模块流程、导入导出、UI Test 或数据安全。

每个 P0 Story 必须能独立验收，不接受“只写 UI、后续再接数据”的完成状态。确实需要分阶段时，Story 名称必须标注 `Mock` 或 `Persistent`。

### 1.2 全局技术约束

- 纯 iOS 原生实现：SwiftUI 为主，UIKit 只做系统能力或性能补位。
- Deployment Target：iOS 17.0。
- 本地数据库：SwiftData。Domain 层不得依赖 SwiftData。
- 视觉基线：参考 `style.md` 和 `docs/design/*.html`，但必须转为 iOS 原生组件和 token。
- iOS 17-25 使用 SwiftUI Material；iOS 26+ 通过封装层采用 Liquid Glass，业务页面不得散落系统版本判断。
- 不引入第三方 UI 框架；首版默认不引入第三方数据库、图表库或 CSV 库。
- 用户可见文案进入 `Localizable.xcstrings`；首版中文，key 使用语义化英文。
- 任何交易金额不写入日志，不接账号、广告或默认云上传。

## 4. Backlog 明细

### E01 Xcode 工程与架构骨架

#### US-001 创建 SwiftUI iOS 工程

优先级：P0  
估算：2  
依赖：无

需求：

- 创建 iOS App 工程，语言 Swift，UI SwiftUI。
- Deployment Target 设置为 iOS 17.0。
- 创建 `NumiApp.swift`、`AppEnvironment`、基础目录结构。

验收：

- App 可在模拟器启动。
- 首页显示临时 RootShell 骨架。
- 工程无第三方 UI 框架。

#### US-002 建立模块目录和命名规范

优先级：P0  
估算：1  
依赖：US-001

需求：

- 建立 `DesignSystem`、`Data`、`Domain`、`Features`、`Services`、`PreviewSupport`。
- 每个目录添加 README 或职责说明文件，说明职责边界。

验收：

- 目录结构与技术方案一致。
- Feature 不直接依赖其他 Feature 的内部类型。

#### US-003 建立 Debug Component Catalog 入口

优先级：P1  
估算：2  
依赖：US-001、US-002

需求：

- Debug 构建下可打开 `ComponentCatalogView`。
- Release 构建不暴露入口。

验收：

- Debug 可从设置页或启动参数进入。
- Release 编译中入口不可见。

#### US-004 建立构建与验证脚本

优先级：P0  
估算：2  
依赖：US-001、US-002

需求：

- 建立 `scripts/verify.sh`。
- 脚本至少执行 Swift Package 单元测试、Xcode build、Xcode UI Test 可用性检查。
- 若 Xcode 工程由脚本生成，验证脚本必须先执行工程生成。

验收：

- 新开发者拉取仓库后可通过一条命令完成基础验证。
- 验证失败时输出明确失败阶段。
- CI 或本地终端都可运行，不依赖个人 Xcode DerivedData 路径。

优先级：P0

验收：

#### US-023 交易写入与账户余额一致性

优先级：P0  
估算：5  
依赖：US-020、US-021

需求：

- 新增、编辑、软删除、恢复交易时同步更新账户余额。
- 转账必须同时影响转出账户和转入账户，但不进入收入/支出统计。
- 编辑交易时采用“冲销旧交易，再应用新交易”的一致性策略。

验收：

- 单元测试覆盖支出、收入、转账、编辑金额、编辑账户、删除、恢复。
- 任一写入失败时账户余额不得半更新。
- 余额计算逻辑不写在 View 或 ViewModel 中。

#### US-024 SwiftData 迁移与恢复点基线

优先级：P0  
估算：5  
依赖：US-020、US-021

需求：

- 建立 Schema V1 版本定义。
- 建立导入前恢复点接口。
- 准备至少一个 V1 样本数据文件用于迁移测试。

验收：

- SwiftData container 可用内存模式跑测试。
- 导入失败后可以恢复导入前状态。
- 后续新增字段有明确迁移入口。

## 7. Definition of Done

每个 Story 完成必须满足：

- 功能满足验收标准。
- 有 SwiftUI Preview 或页面可运行路径。
- 涉及业务规则时有单元测试。
- 涉及 UI 流程时有最小 UI 测试或人工验证说明。
- 不新增第三方 UI 框架。
- 不写死用户可见文案，使用本地化 key。
- 不在业务页面散落颜色 hex 或系统版本判断。
- 对应设计稿已完成原生化差异记录。
- 涉及持久化时通过重启验证。
- 涉及隐私时确认 App switcher 截图不泄露金额。
