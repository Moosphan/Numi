## 1. 背景与目标

### 1.1 产品定位

Numi 是一款纯本地、无广告、低摩擦、审美克制的个人记账 App。它参考 Cookie 记账的核心体验：快速记一笔、清晰明细、趋势洞察、预算管理、订阅分期、多账本、多币种和强自定义；同时把“数据只在本机、用户完全可控”作为第一差异点。

产品不做理财推荐、不做社区、不做导流、不做广告，不把用户账单数据上传到自有服务器。

### 1.2 一句话目标

让用户在 5 秒内完成一笔日常记录，在 30 秒内理解本月钱花在哪里，并能长期安心保存自己的财务生活轨迹。

### 1.3 成功标准

- 新用户首次打开后 2 分钟内完成第一笔支出记录。
- 高频手动记账路径不超过 4 次关键操作：选择分类、输入金额、可选备注/账户、保存。
- 离线状态下所有核心能力可用，包括记账、查询、预算、趋势、订阅分期、导入导出。
- 数据默认保存在本机数据库，导出文件可被用户独立保存和迁移。
- UI 组件库能支撑后续同风格 App 复用，避免页面级硬编码样式。

### 2.3 信息架构观察

Cookie 的核心结构偏“三页主导航”：

- 明细：按月份聚合的账单流水，是用户最常回看的主视图。
- 趋势：收支概览、分类分布、支出明细等数据洞察。
- 我的：设置、资产、预算、订阅分期、主题、数据同步等能力入口。

关键交互模式：

- 底部 Tab 常驻，当前 Tab 使用浅色胶囊高亮。
- “记一笔”是悬浮按钮，放在右下安全区附近，始终可触达。
- 明细页顶部是月份标题与日期范围，左右切换月份。
- 明细页顶部用两张大色块概览支出和收入。
- 流水按日期分组，每个日期右侧显示当天支出/收入小计。
- 记账页使用分类图标网格作为背景主选择，金额面板居中浮层出现。
- 趋势页由多张浅色卡片组成：月概览、支出分布、支出明细等。

### 2.4 视觉语言观察

Cookie 的视觉记忆点：

- 浅色背景，大面积留白，轻卡片，少边框。
- 柔和色块表达财务语义：支出淡蓝、收入淡紫或浅绿、结余淡红、记录次数淡灰蓝。
- 图标大量使用 Emoji，降低冰冷感，也减少传统财务 App 的压迫感。
- 圆角明显但不过分，卡片与按钮以柔和胶囊为主。
- 字体层级简洁：金额粗体、分类中粗、备注和账户弱化。
- 图表不追求复杂可视化，偏列表进度条和低密度统计。

需要注意：Cookie 的视觉很强，但不能逐像素复刻。Numi 应借鉴“轻、快、温和、清晰”的方法，而不是复制品牌名、图标资产、截图布局细节或营销表达。

### 2.5 用户评价洞察

高频正向反馈：

- 简洁、好看、无广告。
- 记账路径顺手，容易坚持。
- 功能丰富但没有明显压迫感。
- 适合新手，也能满足深度用户。
- 预算、趋势、订阅分期、小组件、快捷指令提升了习惯养成。

高频需求和痛点：

- 预算希望支持周预算、分类周预算、剩余额度顺延、超支后重新平均。
- 退款希望冲减原支出，而不是简单记为收入。
- 组合支付希望一笔订单支持多个账户付款。
- 资产希望支持银行/信用卡子账户、共享额度、多币种账户。
- 固定支出和循环费用希望更灵活，如指定周几、自动转账。
- 报销希望可选择是否计入预算或收支。
- 趋势模块希望可隐藏、排序、支持更自由时间范围。
- 图标希望可隐藏、缩放、排序、使用不同风格。
- 导入导出、数据恢复、会员状态、AI 次数等状态需要更清晰的反馈。
- 部分用户仍期待 Android 版本，但前提是无广告和保持体验。

### 2.6 对 Numi 的启发

可直接借鉴的产品原则：

- 高频路径极短，低频能力渐进展开。
- 数据呈现要“看一眼知道状态”，不要堆指标。
- 分类图标和主题是情绪价值，不只是装饰。
- 预算不是单一月额度，而是用户每天/每周做消费决策的工具。
- 订阅、分期、报销、退款、转账是中国用户真实高频财务场景。

Numi 应形成的差异：

- 纯本地优先：不依赖云，不用账号，不上传账单。
- 数据所有权清晰：本地数据库、加密备份、CSV/JSON 导出、可恢复快照。
- 可解释的智能：MVP 不接云端 AI；后续可接本地 OCR/本地规则识别，且默认关闭。
- UI 组件库前置：把审美做成可复用系统，而不是散落在页面里。

## 3. 用户与场景

### 3.1 目标用户

核心用户：

- 18-35 岁，使用手机管理日常生活开销。
- 喜欢简洁好看的工具，不愿被广告、理财产品和复杂表单打扰。
- 有预算意识，但未必是财务专业用户。
- 注重隐私，希望账单数据不上传服务器。

进阶用户：

- 留学生、跨境差旅者，需要多币种。
- 有信用卡、花呗、分期、订阅服务、报销场景的人。
- 长期记账用户，需要导入导出、历史检索和趋势复盘。

### 3.2 核心用户故事

- 作为新用户，我希望不用注册就能开始记账。
- 作为日常用户，我希望吃完饭或购物后 5 秒内记一笔。
- 作为补账用户，我希望能快速选择昨天、上周或自定义日期。
- 作为预算用户，我希望知道本周还能花多少钱。
- 作为订阅用户，我希望 App 自动提醒并记录固定扣费。
- 作为报销用户，我希望公司垫付不影响个人预算。
- 作为隐私敏感用户，我希望 App 进入后台后自动模糊，并能用 Face ID 解锁。
- 作为迁移用户，我希望能导入 CSV，并随时导出完整数据。

## 6. 核心功能需求

### 6.1 快速记账

目标：把最常用的记账动作压缩到最少成本。

默认流程：

1. 点击悬浮“记一笔”。
2. 选择支出/收入/转账。
3. 点击分类。
4. 输入金额。
5. 可选填写备注、日期、账户、标签。
6. 保存或“再记一笔”。

需求：

- 分类网格支持 Emoji、内置线性图标、自定义图片三类图标源。
- 记账面板应支持加减乘除简单计算。
- 日期按钮默认今天，支持昨天、前天、日历选择。
- 账户默认使用上次账户，也可按分类记忆默认账户。
- 备注支持历史联想。
- 保存后给出轻反馈，不打断连续记账。
- “再记一笔”保留日期和账户，可选择是否保留分类。

验收：

- 新建普通支出从打开面板到保存不超过 5 秒。
- 不填写备注也能完成记录。
- 键盘误触删除有明确反馈。

### 6.2 明细

需求：

- 默认按自然月展示，支持自定义账期起止日。
- 顶部展示支出、收入、结余。
- 按日期分组，每组展示当天支出和收入小计。
- 记录项展示分类图标、分类名、备注、时间、金额、账户。
- 支出和收入金额颜色可配置，但默认保持低饱和语义色。
- 支持左滑编辑、复制、删除、退款、报销。
- 支持搜索备注、金额、分类、账户、标签、商家。
- 支持筛选：时间、类型、分类、账户、标签、是否报销、是否订阅/分期。

验收：

- 10000 条记录下滚动流畅。
- 删除前二次确认，支持撤销。
- 筛选状态可清晰看到并一键清空。

### 6.3 分类

需求：

- 支持支出、收入分类。
- 支持两级分类，预留三级分类数据结构但 MVP 不暴露。
- 分类支持图标、颜色、排序、隐藏、预算绑定。
- 子分类可在记账面板中直接展示，也可折叠在父分类下。
- 用户可关闭大图标显示，切换为紧凑列表。

默认支出分类：

- 餐饮、交通、购物、日用品、住房、医疗、娱乐、学习、数码、旅行、人情、宠物、运动、美容、订阅、其他。

默认收入分类：

- 工资、副业、奖金、报销、退款、投资收益、礼金、其他。

### 6.4 账户与资产

需求：

- 账户类型：现金、储蓄卡、信用卡、微信、支付宝、虚拟账户、负债、其他。
- 账户支持初始余额、当前余额、币种、是否计入总资产、是否隐藏。
- 支持账户间转账。
- 支持信用卡额度、账单日、还款日、已用额度。
- 支持银行子账户和多币种子账户，解决同一银行多个币种拆散的问题。

MVP 简化：

- 首版只做账户余额和转账。
- 信用卡额度和子账户作为 V1.1。

### 6.5 预算

需求：

- 支持月预算、周预算、分类预算。
- 月预算可自动拆为周预算，但允许排除房租、水电等固定大额分类。
- 支持剩余额度顺延到下月/下周。
- 支持超支后按剩余天数重新计算每日可用预算。
- 报销和退款可配置是否影响预算。
- 洞悉页展示预算进度、剩余金额、日均建议。

验收：

- 用户能回答“本周还能花多少钱”。
- 固定支出不会强行污染每周日常预算。
- 预算逻辑有说明，不让用户猜。

### 6.6 订阅、分期与循环

订阅需求：

- 记录订阅名称、金额、周期、扣费账户、下次扣费日、分类。
- 支持提前提醒。
- 支持到期自动生成账单或待确认生成。

分期需求：

- 记录总金额、期数、每期金额、首期日期、还款账户、手续费。
- 每期可独立标记已记账、已还款。
- 支持提前结清。

循环记账需求：

- 支持每天、每周、每月、每年、自定义周几。
- 支持自动入账或提醒确认。
- 支持暂停、跳过本期、结束。

### 6.7 报销与退款

报销需求：

- 记录可标记为待报销。
- 可设置报销方、预计报销日期、报销状态。
- 可选择是否计入个人预算。
- 报销到账可冲减原支出或记为收入，默认推荐冲减原支出。

退款需求：

- 从原支出记录发起退款。
- 退款金额可小于等于原金额。
- 趋势中默认冲减原分类支出。
- 记录保留退款链路，方便对账。

### 6.8 趋势与洞悉

需求：

- 时间范围：本周、本月、本年、自定义、自定义账期。
- 概览：支出、收入、结余、记录次数、日均支出。
- 分布：分类排名、账户支出、标签支出。
- 趋势：按日/周/月折线或柱状。
- 明细：Top 分类、Top 商家、异常支出。
- 模块可排序、隐藏、恢复默认。

视觉要求：

- 首屏只放 2-3 个最关键模块。
- 图表优先可读性，不追求复杂炫技。
- 颜色不依赖单一红绿区分，兼顾色弱。

### 6.9 数据导入导出

需求：

- 导出完整数据：JSON。
- 导出表格：CSV。
- 导入 CSV：支持字段映射。
- 本地备份：加密备份包，包含数据库、图标、附件、设置。
- 支持自动备份提醒，但备份文件由用户选择保存位置。

纯本地原则：

- 不默认上传。
- 不默认注册。
- 所有导出行为由用户主动触发。
- 备份文件格式公开说明。

### 6.10 隐私与安全

需求：

- Face ID / Touch ID / 系统密码解锁。
- App 切后台自动模糊。
- 可设置启动即锁、离开 1/5/15 分钟后锁。
- 可隐藏金额显示，进入“隐私浏览”模式。
- 本地数据库可选加密。

## 7. 非功能需求

性能：

- 冷启动目标小于 1.5 秒。
- 10000 条记录明细列表稳定 60fps。
- 趋势统计在 50000 条记录内 1 秒内完成或显示渐进加载。

可靠性：

- 数据写入使用事务。
- 删除和批量修改支持撤销窗口。
- 导入前自动创建恢复点。

可访问性：

- 支持动态字体。
- 可操作控件最小触控区域 44x44 pt。
- 金额、预算、趋势不只靠颜色表达状态。
- 支持 VoiceOver 基础读法。

国际化预留：

- 金额格式、币种、小数位、日期格式可配置。
- 首版中文，数据结构不绑定中文文案。

## 8. 本地数据模型草案

核心实体：

- Ledger：账本。
- Transaction：交易记录。
- TransactionSplit：组合支付或拆分记录。
- Category：分类。
- Account：账户。
- Budget：预算。
- RecurringRule：循环规则。
- Subscription：订阅。
- InstallmentPlan：分期计划。
- Reimbursement：报销。
- Refund：退款。
- Tag：标签。
- Merchant：商家。
- Attachment：附件。
- ExchangeRate：汇率。
- AppSettings：设置。

Transaction 关键字段：

- id。
- ledgerId。
- type：expense / income / transfer。
- amount。
- currency。
- categoryId。
- accountId。
- targetAccountId：转账时使用。
- occurredAt。
- note。
- merchantId。
- tags。
- reimbursementId。
- refundOfTransactionId。
- recurringRuleId。
- installmentPlanId。
- createdAt / updatedAt / deletedAt。

数据设计原则：

- 所有记录软删除，便于恢复。
- 金额以整数最小货币单位保存，避免浮点误差。
- 汇率保存交易当时快照，历史统计不被未来汇率改变。
- 附件只存本地文件引用和校验信息。

## 9. UI 组件库与设计系统

### 9.1 设计风格关键词

- 轻：界面减少装饰，优先内容。
- 软：低饱和色、浅背景、温和阴影。
- 准：金额、日期、状态一眼明确。
- 稳：布局密度可长期使用，不做一次性营销感。
- 可爱但不幼稚：允许 Emoji 和轻松文案，但财务状态必须严谨。

### 9.3 色彩方向

默认主题建议：

- 页面背景：暖白或轻灰，不用纯白压眼。
- 主强调：柔和薄荷绿，用于保存、当前状态、主按钮。
- 支出：低饱和蓝或蓝灰，不使用高刺激红色表达普通支出。
- 收入：低饱和紫或绿。
- 负结余/超支：柔和红。
- 信息卡背景：使用非常浅的语义色块。

约束：

- 不让整个 App 读成单一紫色、单一蓝色或单一米色。
- 财务危险状态可以用红，但避免全局红色焦虑。
- 深色模式后续支持，MVP 可先完成浅色主题。

### 9.5 组件设计原则

- 组件必须消费 semantic token，不直接写页面颜色。
- 所有图标按钮必须有可访问标签。
- 卡片不能层层嵌套，重复信息项才使用卡片。
- 固定格式元素要有稳定尺寸，避免金额变化导致布局跳动。
- 列表行、金额输入、键盘按键必须适配长文本和大字体。
- 每个组件至少定义 default、pressed、disabled、loading、error 或 empty 等必要状态。

### 9.6 关键页面视觉规格

明细页：

- 顶部月份切换居中，左右箭头位于两侧。
- 收支概览使用两到三张浅色 SummaryTile。
- 记录列表按日期分组，日期标题使用弱绿色或弱强调色。
- 悬浮记账按钮固定右下，避开底部 Tab 和安全区。

记账页：

- 分类选择为主背景，金额面板浮在下半部或中部。
- 键盘按键尺寸固定，不随输入内容变化。
- 支出/收入/转账使用底部或顶部 segmented control。
- 保存按钮使用主强调色，“再记一笔”与“记一笔”并列。

洞悉页：

- 首屏优先月概览和预算状态。
- 分布使用横向进度条列表，最多展示 Top 5，更多折叠。
- 模块菜单提供隐藏、排序、说明。

我的页：

- 设置按“数据、记账、外观、安全、关于”分组。
- 不使用营销卡片堆叠，不出现广告位。

## 10. 关键流程

### 10.1 首次启动

1. 展示隐私承诺：本地保存、无账号、可导出。
2. 选择默认币种。
3. 选择是否创建默认账户。
4. 进入明细页，引导点击“记一笔”。

原则：不做长 onboarding，不拦截用户开始记账。

### 10.2 记一笔

1. 用户点击悬浮按钮。
2. 选择分类。
3. 输入金额。
4. 保存。
5. Toast 显示成功，并提供撤销或再记一笔。

### 10.3 月度复盘

1. 用户进入洞悉。
2. 查看月概览和预算状态。
3. 点击分类分布进入该分类明细。
4. 可调整预算或标记异常记录。

### 10.4 数据迁移

1. 用户进入我的 > 数据。
2. 选择导入 CSV。
3. 系统创建恢复点。
4. 用户映射字段。
5. 预览前 20 条。
6. 确认导入。
7. 展示导入结果和错误明细。

## 12. 风险与约束

产品风险：

- 功能面过宽，MVP 容易失焦。必须优先保证快速记账和本地数据可靠。
- 过度追随 Cookie 会导致缺少自身风格。需要通过本地优先和组件化审美体系建立差异。
- 预算、退款、报销、分期的统计逻辑复杂，必须先写清规则再开发。

设计风险：

- Emoji 图标可爱但可能显得不专业，必须提供紧凑/无图标模式。
- 浅色低对比设计容易牺牲可访问性，文本与状态色需过对比检查。
- 图表模块过多会变成信息噪音，必须允许隐藏和排序。

技术风险：

- 本地数据库迁移必须谨慎，否则长期用户数据风险高。
- 导入导出字段映射复杂，需要明确错误处理。
- 加密数据库、附件、备份包会增加恢复难度，需要清晰文档。
